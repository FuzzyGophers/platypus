# Uniden SDS150 (BCDx36HP family)

The current reference device. Everything Platypus knows about it is encoded in the SDS150
`SdCardProfile`
([`crates/platypus-core/src/device/sds150.rs`](../../crates/platypus-core/src/device/sds150.rs));
the rest of the core, FFI, and app stay model-agnostic. Adding a radio is a new
`device/<model>.rs` + a `register()` line; see the [radios index](README.md).

Figures below tagged **(observed)** were measured directly from a real SDS150 card's
database (the byte-exact `samples/` fixtures + the owner's card); ones tagged **(spec)**
come from Uniden's published specs/manual.

## Identity

- `TargetModel` = **`BCDx36HP`**, `FormatVersion` = **`1.00`**: the SD-card *file-format*
  version (distinct from the spec-*document* version, V1.03).
- Serial id = **`SDS150GBT`** (the phase-2 `MDL` handshake).
- `BCDx36HP` is a *family* tag shared across Uniden's DMA scanners (SDS100/SDS200,
  BCD325P2, BCD436HP…), so siblings are largely a profile away — the format is the same;
  limits and a few constants differ.

## Reception & scanning capabilities

- **Frequency coverage:** ~**25–1300 MHz** with the usual cellular gaps (spec).
  The loaded database spans **26.1 MHz – 1298.0 MHz** across ~203k conventional
  frequencies (observed).
- **Conventional modulations** (observed, `C-Freq` field 7): **NFM** (most common), **FM**,
  **AM**, and **AUTO**.
- **Trunked system types** the database/format represents (observed, `Trunk` field 7,
  with rough share): **MotoTrbo** (Motorola DMR/TRBO), **LTR**, **P25Standard** (P25),
  **NXDN**, **DmrOneFrequency** (one-freq DMR), **Motorola** (Type I/II analog), **EDACS**.
  P25 (incl. Phase II/TDMA), Motorola, EDACS, LTR and analog decode on a stock SDS150;
  **DMR / NXDN / ProVoice audio decode require Uniden's paid upgrades** on the device —
  the *file format* carries all of these system types regardless (spec).
- **Talkgroup audio mode** (observed, `TGID` field 7): **ALL** (overwhelmingly), plus
  **ANALOG** / **DIGITAL** to pin a talkgroup's mode.
- Per-channel attributes the format carries: tone/`NAC`/color-code squelch, a
  RadioReference **service-type** code (see below), and the per-record **Avoid** flag.

## The two device interfaces (both officially documented)

1. **File format — the SD card (bulk programming, the heart of the app).** Tab-delimited
   ASCII, CRLF, one sentence per line (`Command \t param \t param`), sub-params split by
   `=` / `/` / `,`. Relational and **geo-tagged** (lat/lon/range on the geo records),
   which is what makes the location-first model essentially free. Validated byte-exact
   against a real card.
2. **Serial protocol — live control (phase 2, optional).** ASCII, comma-delimited,
   `\r`-terminated. Status/control only; bulk programming is the file format above. See
   the [serial protocol](#serial-protocol-live-control) section for the command inventory.

## SD-card data model (record vocabulary)

The schemas Platypus validates (`sds150.rs::SCHEMAS`). Field counts + column indices are
**cross-checked against the Uniden File Spec V2.00** `Format:`/`Parameter:` blocks — every
record below matches (`T-Freq` is the one version delta; see the note under the table):

| Record | Role | Geo | Notes |
|---|---|---|---|
| `TargetModel` / `FormatVersion` / `DateModified` | file header | — | identity (`BCDx36HP` / `1.00`) |
| `StateInfo` (5) / `CountyInfo` (4) | geo master (`hpdb.cfg`) | — | Country→State→County index |
| `Conventional` (15) | conventional system | — | tech in field 6 |
| `C-Group` (11) | conventional department | ✓ | shape: **Circle** (dominant) or **Rectangles** |
| `C-Freq` (18) | conventional **channel** | — | freq@5, mode@6, tone@7, service-type@8 |
| `Trunk` (22) | trunked system | — | tech in field 6 (P25/DMR/NXDN/…) |
| `Site` (20) | trunk site (control channels) | ✓ | per-site geo/range |
| `T-Group` (10) | trunk department | ✓ | usually one per county |
| `TGID` (17) | trunked **talkgroup** | — | tgid@5, mode@6, service-type@7 |
| `T-Freq` (9) | site control/voice freq | — | freq@5, **LCN@6**, avoid@4; scaffolding, not a "channel" |
| `AreaState` / `AreaCounty` (3) | area tags | — | **HPDB only**, dropped in favorites |
| `Rectangle` (6) | rectangular coverage bound | — | for `Rectangles`-shaped groups |
| `BandPlan_P25` (50) | P25 band plan | — | only some sites; see dialect |
| `DQKs_Status` (102) | quick-key enable map | — | favorites-only; **100** key slots |
| `LM` (9) / `LM_Frequency` (4) | location-monitor data | ✓ | lat@7/lon@8 |

Hierarchies: **Conventional → C-Group → C-Freq** and **Trunk → Site/T-Group → TGID**
(plus `T-Freq` under sites). Geo lives on `Site`/`C-Group`/`T-Group` (`lat`, `lon`,
`range` miles, `shape` = the `Location Type` field) — the basis for county placement and
radius selection. **Avoid = field 4** on every system / department / channel record.

Spec-validated details worth recording:
- **`T-Freq` field 6 is the LCN** (Logical Channel Number); field 4 is `Reserve(Avoid)`,
  which the spec says is *always* `Off` — so the favorites dialect forcing col4=`Off` is
  universal, while forcing the LCN (col6) to `0` is a **control-channel** heuristic (real
  voice channels carry non-zero LCNs). The V2.00 spec enumerates 8 `T-Freq` fields; observed
  `1.00` cards carry **9** (the trailing field isn't in the V2.00 doc: a format-version
  delta, preserved verbatim by the byte-exact round trip).
- **`BandPlan_P25` is omissible** unless a site designates a **non-standard** band plan
  (spec: *"If Band Plan is undesignated, `[BandPlan_P25]` is omissible"*) — standard sites
  derive from Band Coverage. Platypus's omit-by-default (`--bandplan` opt-in) is spec-correct.
- **`C-Freq` field 7** is the **Audio Option** union (`null`=All / `Tone=` / `NAC=` /
  `ColorCode=` / `RAN=` / `Area=`); **service type is `FuncTagId`** at `C-Freq`@8 / `TGID`@7.

## Favorites dialect (a different dialect, not a subset)

Same records as the HPDB, but:
- blank `MyId`/`ParentId` (identity columns), drop `AreaState`/`AreaCounty`;
- fill scan-setting field defaults a working favorites populates;
- a few constant overwrites: `T-Freq col6 = 0` is a **validated-but-not-universal**
  heuristic (it let SAFE-T lock its control channel; other systems keep non-zero LCNs);
- synthesize exactly one `DQKs_Status` per system;
- **no `BandPlan_P25` by default**: adding one where it isn't needed stops a trunk
  locking; it's `--bandplan` opt-in.
- **Avoid flag = field 4** on every system / department / channel record (`"On"` = skip,
  `"Off"` = scan). The whole RadioReference database ships `Off`; avoids only ever come
  from a user's prior programming (e.g. an imported Sentinel list).

## `f_list.cfg`: the `F-List` record (per-list settings)

The favorites **index** file `favorites_lists/f_list.cfg` holds the model header plus one
`F-List` line per favorites list. Its fields, transcribed from the **Uniden File
Specification V2.00** (`SDS_Series_File_Specification_V2_00.pdf`, the `F-List` `Format:` /
`Parameter:` block):

| idx | field | values |
|---|---|---|
| 0 | `F-List` | command |
| 1 | UserName | list display name |
| 2 | Filename | slot file (`f_NNNNNN.hpd`) |
| 3 | LocationControl | `On`(=Yes) / `Off`(=No) |
| **4** | **Monitor** | **`On`(=Monitor) / `Off`** |
| 5 | Quick key | `Off` / `0`–`99` |
| 6 | NumberTag | `Off` / `0`–`99` |
| 7–16 | Startup key 0–9 | `On` / `Off` |
| 17–116 | S-Qkey_00–99 | `On` / `Off` |

A real card's record is **118 fields** (one more than the V2.00 spec enumerates), so the
trailing field is **preserved-but-unidentified**. **Monitor** = the radio's *Select Lists to
Monitor* state: a monitored list is always scanned when active; an un-monitored list is gated
by its quick key (a list with no quick key is also always scanned).

**Write rule (critical):** on any `f_list.cfg` rewrite, Platypus changes **only** UserName
(field 1), slot membership, list order, and **Monitor** (field 4, exposed as a per-list
toggle in the app). **Every other field is carried through verbatim** from the existing
entry: quick key, number tag, startup keys, S-Qkeys, and the unidentified trailing field.
The hardcoded default block ([`Sds150::f_list_entry`](../../crates/platypus-core/src/device/sds150.rs);
Monitor field via `f_list_monitor_field` → `4`) is synthesized **only for a brand-new list**.
This is why a save no longer disturbs a user's quick-key / monitor assignments; earlier it
regenerated the whole block from defaults (Monitor=`On`), flipping every list to Monitor.

## Location features

- Every `Site` / `C-Group` / `T-Group` carries a coverage geometry: center `lat`/`lon`,
  a `range` in **miles**, and a `shape` (**Circle**, dominant; **Rectangles**, rare with
  explicit `Rectangle` bounds). This is what location-first scanning and Platypus's
  county placement / radius selection run on.
- A statewide trunk's talkgroups are organized into per-county `T-Group`s, each
  geo-tagged, so a county or point+radius pulls just that area's talkgroups.

## Service types

Channels carry a numeric **service-type code** (`FuncTagId`; `C-Freq`@8 / `TGID`@7). The
full enumeration is the **Uniden File Spec V2.00 "Service Type" sheet**: 37 codes (with
`5/10/18/19/27/28/35/36` marked `non`/unused) plus Custom `208`–`217`. Platypus maps all of
them, in [`model::service_type_name`](../../crates/platypus-core/src/model.rs) (names) and
the [`ServiceType`](../../apps/PlatypusMac/Sources/PlatypusMac/Catalog/ServiceType.swift)
table (name + SF Symbol + color); unknown/`non` codes fall back to a neutral glyph. Codes
run: 1 Multi-Dispatch, 2 Law Dispatch, 3 Fire Dispatch, 4 EMS Dispatch, 6 Multi-Tac, 7 Law
Tac, 8 Fire-Tac, 9 EMS-Tac, 11 Interop, 12 Hospital, 13 Ham, 14 Public Works, 15 Aircraft,
16 Federal, 17 Business, 20 Railroad, 21 Other, 22 Multi-Talk, 23 Law Talk, 24 Fire-Talk,
25 EMS-Talk, 26 Transportation, 29 Emergency Ops, 30 Military, 31 Media, 32 Schools,
33 Security, 34 Utilities, 37 Corrections. (Earlier empirical guesses had `22`/`23` wrong:
they are Multi-Talk / Law Talk, not Transportation / City Services.)

## Limits

**256** favorites lists · **1 MB** per list · **100** quick keys — the 100 quick keys are
exactly the 100 `DQKs_Status` slots in the favorites format.

## What Platypus has validated on real hardware

- **Byte-exact round-trip** of the card format (the writer safety gate).
- A **conventional** favorites we generated **scanned live traffic** on a real SDS150.
- A **trunked P25** favorites (SAFE-T, location-filtered) **decoded live traffic** (audio)
  once `T-Freq col6 = 0` was applied.
- `DQKs_Status` is a quick-key *preference*, **not** a scan gate (both `On`/`Off` work).
- `f_list.cfg` registration + the `app_data.cfg`-delete + eject flow — no SD errors.

## SD-card layout & device-specific write rules

- **Delete `app_data.cfg` after writing program data**, or the scanner misbehaves on
  resume. Baked into every `card` write path.
- **The favorites-index file is `discvery.cfg`**, misspelled on real cards (no second
  "o"). Stored verbatim.
- Layout: model folder `BCDx36HP`; favorites slots `favorites_lists/f_<NNNNNN>.hpd`; list
  index `f_list.cfg`; resume state `app_data.cfg`.

## Display customization

The on-screen display is customizable via the settings config (a **third file class**, not
HPDB or favorites): `DispOptItems` (which data items show per layout), `DispColors`
(per-element text/background colors), `DisplayOption` (TGID format, color mode), `Backlight`.
The full spec-ready reference (record formats, the 7 layout modes, item-code tables by area,
and the 147-color palette) is in **[`sds150-display.md`](sds150-display.md)**. It's bulk SD
programming, so the same write rules apply (round-trip gate, `fsync`+eject, delete
`app_data.cfg`).

## Serial protocol (live control)

Phase-2, optional — **status + control, never bulk programming** (that's the SD card). From
the **Uniden Remote Command Specification V2.00** (the `BCDx36HP` family; originally
BCD536HP/436HP v1.05). It's a **USB virtual serial port**: on plugging in USB the radio
prompts *"Select USB mode: Mass Storage=`E` / Serial Port=`.`"* — pick Serial Port. Commands
are **ASCII, comma-delimited, `\r`-terminated**; responses echo `CMD,<data>\r` or `CMD,OK\r`.

- **Handshake:** `MDL` → `MDL,SDS150GBT` (matches [`serial_model_id`](../../crates/platypus-core/src/device/sds150.rs));
  `VER` → firmware version. `KAL` = keep-alive (no response).
- **Live status (read):** `STS` (the raw 20-line display + font/mode flags), `GSI`/`PSI` (rich
  **XML** `ScannerInfo`: current system/department/channel/site/frequency/TGID…; `PSI` is
  pushed), `GST` (status + waterfall), `GCS` (battery: `VOLT`/`%`/`CURR`/`TEMP`), `GLT` (get
  lists), `PWF`/`GWF`/`GW2` (FFT waterfall stream).
- **Control (write):** `KEY` (full remote keypad by key code), `QSH` (quick-search hold to a
  frequency), `HLD` (hold system/dept/channel), `NXT`/`PRV`, `JNT` (jump number tag), `JPM`
  (jump mode), `MNU`/`MSI`/`MSV`/`MSB` (menu navigation), `URC` (recording), `POF` (power off).
- **Settings (get/set):** `VOL` (0–15), `SQL` (0–15), `FQK`/`SQK`/`DQK` (favorites/system/dept
  **quick-key enable**, the live twin of the file format's quick keys), `AVD` (avoid:
  1=permanent / 2=temporary / 3=stop), `SVC` (service-type settings), `DTM` (date/time).

The protocol **codec** is implemented (encode commands / parse responses, no I/O) in
[`platypus-core::serial`](../../crates/platypus-core/src/serial.rs) — the piece a future
`serialport` transport drives.

**Standout for Platypus: `LCR` "Get/Set Location and range."** `LCR,<lat>,<lon>,<range>\r`
sets the scanner's location + range **live over serial** (degrees). That makes location-first
a *live* control surface, not just a file-programming trick: pick a point+radius on the map
(or a ZIP) and push it straight to the radio. Combined with `AVD` and `FQK/SQK/DQK`, a phase-2
serial transport enables live re-scoping and avoid toggling without a card write — the
natural companion to the device-class split in
[`docs/architecture.md`](../architecture.md).
