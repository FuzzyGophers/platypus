# Yaesu FT-60R

**Status: supported.** The FT-60R is the second radio class, a clone-image handheld,
registered as a `CloneImageProfile` (the sibling to `SdCardProfile`); its channels are read,
edited, and written over the serial clone cable. This doc is the device reference. Figures
tagged **(spec)** are from Yaesu's official Operating Manual; ones tagged **(CHIRP)** are from the
open-source [`chirp/drivers/ft60.py`](https://github.com/kk7ds/chirp/blob/master/chirp/drivers/ft60.py)
reverse-engineered memory map — verify against that source when implementing.

## Why it's here — the value prop still holds

The FT-60 is a **dual-band FM ham handheld**, not a database scanner: no SD card, no
relational geo-tagged file, no per-channel lat/lon. So the SDS150's "county coverage lands
in the right place automatically" doesn't transfer directly. **But location-first is a
Platypus *interface*, not a file feature** — it still applies here through:

- the **map lens**: plot repeaters/simplex around a point+radius (coordinates come from
  RadioReference / repeater directories, not the radio), then commit the ones in range;
- planned per-selection **knobs** (*what antenna? what's your expected range?*) to shape
  what's worth loading for where you actually are;
- **better organization + labeling** than the incumbent. CHIRP's editor is a raw
  **spreadsheet**: one row per channel, cryptic columns. The whole Platypus thesis (drill
  by place, name things sanely, group meaningfully) is an upgrade in exactly the dimension
  CHIRP is weakest. This is a **memory manager with a good UX**, and a map, layered over a
  radio that's only ever had spreadsheets and radio-to-radio cloning.

So the FT-60 reframes Platypus from "scanner database pruner" toward "location-aware
programming manager for handhelds" — same north star, different substrate.

## Identity & class (spec)

- Dual-band **144 MHz / 430 MHz FM** transceiver; 5 W. Model code **`AH017`** (the clone
  magic; **CHIRP**).
- Programmed by a **clone/serial cable**, not a filesystem. There is **no official PC file
  spec**: Yaesu never published the memory layout; the community map is CHIRP's.

## Specifications (spec — Operating Manual EH017M209)

- **RX:** 108–137 MHz (AM air), 137–520 MHz, 700–999.99 MHz (cellular blocked).
- **TX:** 144–148 & 430–450 MHz (narrower per region/version).
- **Channel steps:** 5 / 10 / 12.5 / 15 / 20 / 25 / 50 / 100 kHz.
- **Power:** 5.0 W (High) / 2.0 W (Mid) / 0.5 W (Low). **Emission:** F2D, F3E. **±5 ppm.**
- **CTCSS / DCS** (with Bell), **EPCS** paging, ARTS, DTMF memories.
- **Memory:** **1000 "standard" channels (000–999)**, **5 Home** (one per band), **50 PMS
  band-edge pairs** (L01/U01–L50/U50), **10 NOAA weather**, **10 memory banks** (a channel
  may live in several banks).

## Device interface — clone over serial

- **Official (spec):** radio-to-radio only, via the optional **CT-27** cable between the two
  MIC/SP jacks. Both radios: hold **MONI** at power-on → set **`F8 CLONE`** → `[F/W]` →
  press MONI on the destination to receive. No per-field PC protocol is documented.
- **PC transfer (CHIRP):** a **PC↔radio clone** over a USB-serial cable (FTDI preferred;
  avoid counterfeit PL2303). It reads/writes the **entire EEPROM image** as one blob — there
  is no incremental "write one channel." Key transport constants:

  | Item | Value |
  |---|---|
  | Baud | **9600** |
  | Image size | **28,617 bytes** (`_memsize`) |
  | Framing | 8-byte header, then **448 × 64-byte blocks** |
  | ACK | `0x06` after each block |
  | Model magic | `b"AH017"` (`_model`) |

## Memory map

> **Facts only.** The offsets, sizes, and protocol constants below are facts about the radio,
> cross-referenced from public references (tagged **(CHIRP)** vs the Yaesu **(spec)**) and
> real hardware. The codec is spec-derived from these facts; no reference's code is copied.
> See [`CREDITS.md`](../../CREDITS.md).

Fixed offsets into the 28,617-byte image (**CHIRP**; treat as the spec-of-record until we
validate against hardware):

| Region | Offset | Notes |
|---|---|---|
| Standard memories 000–999 | `0x0248` | one `struct mem` each |
| PMS band-edge pairs (L/U) | `0x40C8` | 50 pairs, surfaced as ch 1001–1100 |
| Channel names | `0x4708` | **6 bytes** each, custom charset (digits `0x00–09`, `A–Z` `0x0A–23`, space `0x24`, symbols `0x25–3F`) |
| Bank membership | `0x69C8` | 10 banks × 128-byte bitmap (1 bit/channel) |
| Skip flags | `0x6EC8` | 2 bits/channel: `""` / `S` (skip) / `P` (pref.) |
| Set-mode settings | `0x0024` | the menu block (see below) |

Set-mode settings (**CHIRP**; each value is the **low bits** of a dedicated byte, high bits
reserved/preserved — modeled by `settings_specs` in `device/ft60.rs`):

| Setting | Offset | Bits | Values |
|---|---|---|---|
| Auto power-off (`apo`) | `0x24` | full byte | Off, 0.5–12.0 h |
| Time-out timer (`tot`) | `0x25` | `& 0x1F` | Off, 1–30 min |
| RF squelch (`rf_sql`) | `0x28` | `& 0x0F` | Off, S-1…S-8, S-Full |
| Control lock (`lock`) | `0x2B` | `& 0x07` | Key / Dial / PTT combos |
| DTMF delay (`dt_dly`) | `0x2C` | `& 0x07` | 50–1000 ms |
| DTMF speed (`dt_spd`) | `0x2D` | `& 0x01` | 50 / 100 ms |
| ARTS beep (`ar_bep`) | `0x2E` | `& 0x03` | Off / In range / Always |
| Lamp (`lamp`) | `0x2F` | `& 0x03` | Key / 5 sec / Toggle |
| Bell (`bell`) | `0x30` | `& 0x07` | Off / 1 / 3 / 5 / 8 / Continuous |
| Battery saver (`rxsave`) | `0x31` | `& 0x07` | Off / 200 ms … 2 s |

The multi-flag bytes (`0x38`–`0x3E`: weather alert, ARS, beeps, scan resume, LEDs, busy-lockout)
are preserved verbatim, not yet modeled — a later pass.

Per-channel record (`struct mem`, **CHIRP** — bitfields, MSB-first):

```
u8   used:1, unknown1:1, isnarrow:1, isam:1, duplex:4;
bbcd freq[3];                       // ×10000 to Hz, with step corrections
u8   unknown2:1, step:3, unknown2_1:1, tmode:3;
bbcd tx_freq[3];
u8   power:2, tone:6;               // CTCSS index
u8   unknown4:1, dtcs:7;            // DCS index
u8   unknown5;
u16  unknown5_1:1, offset:15;       // repeater offset
u8   unknown6[3];
```

Enumerations (**CHIRP**): `TMODES = ["", "Tone", "TSQL", "TSQL-R", "DTCS", "DTCS->",
"Tone->DTCS", "DTCS->Tone"]`; `DUPLEX = ["", "", "-", "+", "split", "off"]`;
`STEPS = [5, 10, 12.5, 15, 20, 25, 50, 100] kHz`; `POWER_LEVELS = High 5 W / Mid 2 W /
Low 0.5 W`.

**Validated against real captures (Platypus codec) — read *and* write.** The full per-channel
record is decoded/encoded (round-trip byte-exact): `freq`, `duplex`, `offset`, `tx_freq` (split),
`mode` (FM/NFM/AM), `tmode` sub-kind + CTCSS/DCS value (incl. the cross modes carrying both),
`power`, `step`, name, banks, skip. **The clone-out write path is hardware-proven**: an edited image
(power + offset changes) was written to a real FT-60 and re-captured **byte-identical** except one
volatile "current channel" state byte the radio itself updates (and the trailing checksum) — the
exact "preserve what we don't model" contract holding on write.
- **Name charset** — digits (`0x00–09`), `A–Z` (`0x0A–23`), space (`0x24`), then the symbol block
  `0x25–3F` (`! \` o $ % & ' ( ) * + , - . / | ; / = > ? @ [ u ] ^ _`). Confirmed byte-for-byte
  against hardware: a name walking the symbols in dial order captured back as the exact consecutive
  run `0x25…0x3F`. `o`/`u` are the radio's small-glyph characters and `/` appears twice
  (`0x33`/`0x36`); those are decode-only, every common punctuation mark round-trips on edit.
- **`offset`** (bytes 11–12, big-endian, top bit `unknown5_1`) counts **50 kHz steps** —
  **validated against hardware** at two points: `0x000C` = 12 → 600 kHz (2 m) and `0x0064` = 100 →
  5 MHz (a 70 cm `+` repeater, written to the radio and read back). Simplex channels carry a default
  too; `duplex` gates its use.
- **`power`** index — **validated against hardware**: `0` = High, `1` = Mid, `2` = Low (confirmed on
  the radio's power-set display after writing one channel at each level; the main display shows just
  "Low" for both Mid and Low, but the set screen distinguishes them).
- **PMS band-edge memories** are **100 more identical 16-byte records** contiguous at
  `0x40C8` (`0x0248 + 1000·16`), 50 interleaved lower/upper pairs; decoded + round-tripped.
  Editing them in-app is deferred until a programmed sample confirms the pairing.

## What a Platypus FT-60 profile would take (architecture sketch)

> The trait split that made this land is in
> [`docs/architecture.md`](../architecture.md) — `RadioProfile` base +
> `SdCardProfile` / `CloneImageProfile` siblings, format vs transport. The summary below is
> the "why."

This radio breaks three assumptions baked into today's core — worth deciding on before
committing:

1. **Transport = serial, not a filesystem.** Programming *is* the clone path, so this
   pulls the planned `serialport` crate forward from "phase-2 optional" to "required." A new
   transport trait (open port → clone-in image / clone-out image) sits beside the existing
   card I/O; the app gains a guided "put the radio in F8 CLONE, then…" flow (the one genuine
   UX tax of this radio class).
2. **Binary EEPROM image, not tab-delimited sentences.** The `format` layer
   (`Document`/`Line`, the sentence parser) doesn't apply. Instead a fixed-offset **bitfield
   codec** decodes the 28,617-byte image ↔ a typed model. The **byte-exact round-trip gate
   still works and is still the safety contract** (read image → decode → re-encode → assert
   identical bytes) — which is the right guardrail for a codec we don't fully understand
   (note the several `unknown*` fields: preserve them verbatim, exactly the "never overwrite
   what we don't know" rule we already follow for the SDS150 `F-List` block).
3. **No geo in the radio → Platypus supplies place, the radio supplies channels.** The
   canonical model's `Dataset`/coverage stays the source of location; the FT-60 profile is a
   sink that maps chosen channels → the 1000-slot memory + banks + names. The SD-card-shaped
   methods (record schemas, favorites dialect) already live in `SdCardProfile`, not the
   `RadioProfile` base, so the FT-60 implements a sibling `CloneImageProfile` and skips them —
   likely split into a small transport/format-agnostic core trait plus per-class extensions,
   so "adding a scanner" and "adding a clone-image HT" are both just a new `device/*.rs`.

Net: the core's **typed model, extraction/filters, map lens, and round-trip discipline are
reusable**; the **file-format and card-I/O layers are SDS-specific** and get a sibling, not
a rewrite.
